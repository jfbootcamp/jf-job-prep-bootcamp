1. 콜백 함수
    1) 비동기 처리 후, 후속 작업을 수행하기 위해 사용하는 함수 
    2) 비동기 작업이 끝난 이후 실행할 코드를 전달 할 때 사용됨 
    3) 비동기 함수는 작업이 완료된 후 콜백 함수를 호출하여 데이터를 전달함 

2. 콜백이 필요한 이유 
    1) 비동기 함수 내부에서 데이터를 리턴하면,
       비동기 작업이 완료되기 전에 리턴되므로 결과를 받을 수 없음
    2) 따라서 비동기 작업이 끝났을 때 실행할 콜백 함수를 넘겨줘야 함    

3. 콜백 지옥(헬)
    1) 방식으로 비동기 작업을 여러 번 수행하면, 코드가 계속 중첩되어 가독성이 떨어짐
    2) 깊은 중첩으로 인해 코드 관리와 디버깅이 어려워지는 현상
    3) 이런 콜백 지옥 문제를 해결하기 위해 Promise가 도입됨 

4. Promise
    1) 콜백 헬 문제를 해결하기 위해 ES6에서 도입됨 
    2) Promise는 비동기 작업의 상태를 관리하는 객체 
    3) 작업 진행중: Pending
    4) 작업 성공: Fulfilled 
    5) 작업 실패: Rejected

5. Promise 동작 원리 
    1) Promise 객체 생성 시 콜백 함수를 인자로 전달 
    2) 콜백 함수는 2개의 인자를 가짐 
        - 성공 시 호출 될 함수(resolve)와 실패 시  호출할 함수(rejected)
    3) 비동기 작업을 수행하는 코드는 이 콜백 함수 내부에 작성됨       
    4) 비동기 작업이 성공적으로 끝나면 콜백 함수의 첫번째 인자인 resolve를 호출하여 
       Promise 상태를 fullfilled로 변경하고 결과 값을 전달함 
    5) 비동기 작업 중 에러가 발생하면 두번째 인자인 reject를 호출하여 
       Promise 상태를 rejected로 변경하고 에러 객체를 전달함       

6. Promise의 then, catch, finally
    1) then
        - Promise의 상태가 Fulfilled(성공)가 되었을 때 호출됨 
        - then 메서드는 연쇄적으로 (chaining) 사용할 수 있으며,
          이전 then의 결과를 다음 then에서 전달받아 사용 가능함 
    2) catch
        - Promise의 상태가 rejected(실패)가 되었을 때 호출됨
        - then으로 연결된 작업 중 어떤 단계에서든 에러가 발생하면 가장 가까운 catch가 호출됨
    3) finally
        - Promise가 fullfilled(성공)이든 rejected(실패)이든 상태와 무관하게 무조건
          실행되는 메서드 
        - 주로 로딩 상태표시, 리소스 정리 등의 마무리 작업에 사용됨 
        - finally는 인자를 전달받지 않으며, Promise 체인의 마지막 단계에 주로 사용됨
