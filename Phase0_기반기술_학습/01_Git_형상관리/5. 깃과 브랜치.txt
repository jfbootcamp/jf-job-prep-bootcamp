1. 브랜치(branch)
    1) 나무가지 
    2) 나무가 가지에서 새 줄기를 뻗듯이 여러 갈래로 퍼지는 데이터 흐름을 가리킴 

2. 브랜치 필요 이유 
    1) 제품 출시 
        - 고객사마다 추가로 요구하는 내용이 달라질수 있음 
        - 처음에 작업한 저장소(main) 전체를 여러 개 복사해서 고객사마다 이름을 붙인 후 
          버전 관리 따로 하는 방법.

          A => B => C => D

          openAI  :    A => B => C => D => OA  
          google  :    A => B => C => D => GE => GF => GH 
          nvidia  :    A => B => C => D => ND   
          main    :    A => B => C => D => E 

        - 저장소마다 버전 관리를 따로 하는 방법           

3. 브랜치 기능 살펴보기 
    1) 깃으로 버전 관리를 시작하면 기본적 main 브랜치가 만들어짐.
    2) 사용자가 커밋할 때마다 main 브랜치는 어떤 게 최신 커밋인지 정보가 들어 있음.
        - 브랜치는 커밋을 가리키는 포인터와 비슷함 

    3) 새 브랜치 생성 (분기한다)
        - 기존 파일은 main 브랜치에 그대로 유지하면서,
          새 브랜치에서 기존 파일 내용을 수정하거나 새로운 기능을 추가 할 수 있음.
        - main 브랜치에서 새 브랜치를 만드는 것 => 분기한다 (branch) 

    4) 병합한다 (merge)
        - 새 브랜치에서 작업을 다 끝났다면 새 브랜치에 있던 파일을 
          원래 main 브랜치에 합칠 수 있음.
        - 분기했던 브랜치를 main 브랜치에 합치는 것을 병합한다(merge)고 함.   

4. 새 브랜치 만들기 
    1) 명령어
        - git branch <브랜치명>   
    2) main 앞에 *가 표시된 것
        - 여러 브랜치 가운데 현재 작업하는 브랜치를 나타냄             

5. 새 커밋을 추가하기 

6. 브랜치 전환하기 -- git switch 
    1) 브랜치가 여러 개라면 브랜치마다 서로 다른 커밋을 만들 수 있음.
    2) 브랜치마다 오가면서 작업 할 수 있어야 함 
    3) 현재 브랜치에서 다른 브랜치로 이동하는 것 
    4) 브랜치 간에 이동할 때 checkout 하는 것임 
        - 브랜치를 바꾸는 것을 '체크아웃한다'라고도 말함 

7. 전환한 브랜치에서 커밋하기         

8. 브랜치와 커밋의 관계 알아보기 
    1) git log --oneline --branches    
    2) git log --oneline --branches --graph

9. 브랜치 간의 차이점 살펴보기 
    1) git log 브랜치1..브랜치2    
        - 브랜치 이름 사이에 마침표 2개(..)를 넣는 명령 
        - 브랜치 이름과 마침표 사이는 공백 없이 입력함 
        - 마침표 왼쪽에 있는 브랜치를 기준으로 오른쪽 브랜치와 비교 함 
    2) git log main..openai
        - main 브랜치에는 없고 openai 브랜치에만 있는 커밋을 보여줌 
    3) git log openai..main
        - openai 브랜치를 기준으로 main 브랜치와 비교함
        - openai 브랜치에는 없고 main 브랜치에만 있는 커밋을 보여줌
